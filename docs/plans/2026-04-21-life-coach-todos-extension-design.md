## 🔒 스코프 잠금 선언 (맨 위 고정)

이 문서는 **1주 timebox**(2026-04-21 월 ~ 2026-04-27 일) 안에 MVP 동작을 목표로 한다.

- **범위**: `todos` + `daily_checkins` 테이블 + CRUD + life-coach에 아침/저녁 액션 2개 + 이번 분기 backlog 초기화 + `task_suggestions` read-only 노출
- **시간**: 7일 (설계 + 구현 + 동작 확인 전부 포함. "1주 기획 + 1주 구현" 아님)
- **실패 조건**: 7일 뒤 MVP 동작 안 하면 스킬 접고 **노션 페이지 1개 + Claude 매일 아침/저녁 대화 루틴**으로 전환. 매몰비용 금지.
- **명시적 제외**: 자동 trigger, health/finance/pantry 통합, 통합 리포트, 건강↔작업 교차분석, task_suggestions↔todos **승격 워크플로우**, 반복(recurrence) todo
- **중간 스코프 확장 유혹 차단**: "이것도 같이…"가 떠오르면 → 아래 Phase 2+ 섹션에만 기록. Phase 1 코드 건드리지 않음.

## 배경

### 문제 statement

지난 2주(2026-04-07 ~ 2026-04-20) 야크 쉐이빙 패턴:

```
에이전트 만들기 (WIP 1)
  → 데이터 보기 (WIP 2, 끝이 없음 — 완료 조건 미정의)
    → 스킬로 먼저 구체화 (WIP 3, 더 모호)
      → 일일 PM 에이전트 구상 (WIP 4, 전부 정지)
```

근본 원인:

- 동시 진행(WIP) 무제한 증가 — 새 할일 발생 시 backlog가 아니라 즉시 WIP 추가
- 완료 정의 모호한 작업(`초안`, `기획`, `스킬 완성`, `데이터 본다`)이 누적
- 상위 목표 확정 부재 → 매일 "중요해 보이는 것"에 끌림
- 외부 추적 시스템 부재 → 진도가 보이지 않음 → 더 많은 일을 시작

### 사용자 니즈

- 매일 아침: "오늘 뭐 할지, 어디까지 할지" 외부 의식으로 고정
- 매일 저녁: "어제/오늘 뭐 했는지 리뷰" + 아침 계획 vs 실제 대조
- todo 스코프: **회사 업무 + 개인 삶(건강/재정 등) 통합 관리**
- Deadline, 우선순위, subtask 등 todo 관리 기본 기능
- AI가 자동 추출한 제안(`task_suggestions`)이 dead storage 되지 않고 **눈에 보일 것**

### 비목표

- 팀용 PM 시스템 (별도 `pm-bot`이 담당 — 이 스킬과 무관)
- "인생 관리 AI 에이전트" 전체 (스코프 팽창 함정 — 단계적 확장만)
- 자동 trigger/알림 (Phase 2+)

## 기존 인프라 분석 (재사용 맵)

| 인프라 | 역할 | 본 스킬에서 재사용 방식 |
|---|---|---|
| `sessions`, `session_content` | CC 세션 raw 기록 (자동, SessionEnd hook) | 저녁 액션에서 폴백 조회 |
| `tasks` (기존) | 과거 작업 정제 기록 (segments, duration_min) | 저녁 액션 "어제 뭐 했어?" 데이터 소스 |
| `task_suggestions` (기존) | AI 코칭이 추출한 할일 제안 (pending/resolved) | **Phase 1: read-only 노출 (참고 섹션)**. 승격은 Phase 2. |
| `projects` (기존) | 프로젝트 묶음 | `todos.project_id`로 재사용 |
| `followup_chains` (기존) | follow-up 추적 | 영향 없음 |
| `work-digest` Step 1-5 | 세션 수집→요약→task 생성 | 저녁 액션이 **세션 내에서 직접 호출** |
| `life-coach` Phase 1-3 워크플로우 | 데이터 준비→코칭→리포트 | **재구성** (아래 섹션 참조) |
| `get_pending_tasks(conn)` db.py 기존 함수 | pending task_suggestions 조회 | **재사용** (새 함수 추가 없음) |

### work-digest 자동화 현황 (중요)

**현재 상태**: work-digest 자동 실행 중단 — pm-bot을 돌리던 OpenClaw 에이전트가 같이 수행하던 구조가 OAuth 이슈로 정지.

**본 설계 영향**:

- 아침/저녁 액션은 원래부터 Claude 세션에서 수동 invoke 설계 (cron 아님) → 직접 영향 없음
- 저녁 액션이 `tasks` 조회 시 비어있으면 → **현재 Claude 세션에서 work-digest Step 1-5 직접 실행**. work-digest SKILL.md가 이미 "사용자 요청이나 다른 스킬의 호출로 실행"을 명시하고 있어서 정상 경로.
- 부수 효과: 이 스킬이 매일 돌면 work-digest도 자연스럽게 매일 실행됨 → 자동화 공백의 사실상 복구

### life-coach 사용 현황 (중요)

**현재 상태**: 기존 `daily` / `weekly` cron도 **pm-bot 에이전트 중단과 함께 dead**. 수동 invoke도 사실상 안 됨. → life-coach 전체가 **실질 dead skill** 상태.

**본 설계 방향**: life-coach **재구성**. 아래 "life-coach 재구성" 섹션 참조.

## Gap 분석

| 축 | 기존 상태 |
|---|---|
| 작업 기록 (retrospective) | ✅ 완비 (work-digest + tasks + session_*) |
| AI 할일 제안 | ✅ 완비 (task_suggestions) — 단 **노출 경로 없음** |
| 프로젝트 묶음 | ✅ 완비 (projects) |
| 건강/식사/재정/식재료 기록 | ✅ 완비 (health_*, finance_*, pantry_*) |
| **사용자 todo 관리 (prospective)** | ❌ **없음** |
| **매일 아침/저녁 의식** | ❌ **없음** |
| **Done 정의 강제** | ❌ **없음** |
| **WIP limit** | ❌ **없음** |
| **AI 제안 노출 경로** | ❌ **없음 (dead storage)** |

이 5개가 Phase 1이 채울 gap.

## 데이터 모델

### `todos` 테이블 (신규)

```sql
CREATE TABLE todos (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    title TEXT NOT NULL,
    done_definition TEXT,
    status TEXT NOT NULL DEFAULT 'backlog',  -- backlog/wip/done/blocked/deferred
    priority INTEGER,                         -- 1(high) ~ 3(low), null 허용
    project_id INTEGER,                       -- 기존 projects 참조
    parent_id INTEGER,                        -- subtask (self-ref)
    category TEXT,                            -- 업무/개인/건강/재정/관계
    quarter TEXT,                             -- "2026Q2" 장기 bucket
    deadline TEXT,                            -- ISO 8601: "2026-04-25" or "2026-04-25T18:00"
    estimated_min INTEGER,                    -- 예상 소요 시간 (분)
    notes TEXT,
    created_at TEXT NOT NULL DEFAULT (datetime('now','localtime')),
    started_at TEXT,                          -- backlog → wip 전환 시각
    done_at TEXT,                             -- wip → done 전환 시각
    deferred_reason TEXT,                     -- deferred 상태일 때 이유
    FOREIGN KEY (project_id) REFERENCES projects(id),
    FOREIGN KEY (parent_id) REFERENCES todos(id)
);

CREATE INDEX idx_todos_status ON todos(status);
CREATE INDEX idx_todos_deadline ON todos(deadline);
CREATE INDEX idx_todos_category ON todos(category);
CREATE INDEX idx_todos_parent ON todos(parent_id);
```

**필드 설계 근거**:

- `status`에 `blocked`, `deferred` 포함 — 단순 done/not done 이분법 아닌 실제 상태 분리. 지난 2주 "안 끝남"의 한 원인이 "지금은 못 하는데 list에 남아 스트레스" → blocked/deferred로 인지 분리.
- `done_definition` — Done 정의 강제 기제. 비어있으면 WIP 진입 금지 (운영 규칙 참조).
- `parent_id` — "cube-admin 기획" 같은 모호 todo를 subtask로 쪼갤 수 있게. 지난 2주 "기획이 안 끝남"의 직접 대응.
- `category` — 업무+개인 통합 관리의 분류 축. TEXT 자유 허용 (정규화는 Phase 2에서 수집된 값 기반 결정).
- `deadline` + `estimated_min` — 마감 임박 판정용. "남은 시간 < 예상 소요" = 빨간불.

**`priority` 사용 규칙 (명시)**:

```
Backlog 정렬 SQL:
  ORDER BY
    CASE WHEN deadline IS NULL THEN 1 ELSE 0 END,  -- deadline 있는 것 먼저
    deadline ASC,                                   -- 임박 순
    CASE WHEN priority IS NULL THEN 99 ELSE priority END ASC,  -- high 먼저
    created_at ASC                                  -- 오래된 것 먼저

아침 액션의 "Backlog top 5"는 이 정렬 기준 상위 5개.
priority 값:
  1 = high (반드시 이번 주), 2 = medium (가능하면), 3 = low (여유 되면)
  null = 분류 안 됨. 정렬 시 가장 뒤로.
```

### `daily_checkins` 테이블 (신규)

```sql
CREATE TABLE daily_checkins (
    date TEXT PRIMARY KEY,                    -- KST 날짜 "2026-04-21"
    morning_wip_ids TEXT,                     -- JSON array: [id1, id2]
    morning_intent TEXT,                      -- 오늘 어디까지
    evening_reflection TEXT,                  -- 하루 끝 자기 기록
    created_at TEXT NOT NULL DEFAULT (datetime('now','localtime')),
    updated_at TEXT NOT NULL DEFAULT (datetime('now','localtime'))
);
```

**필드 설계 근거**:

- `morning_wip_ids` JSON — 오늘의 WIP todo ids (WIP limit 2 강제)
- `morning_intent` — "오늘 어디까지" (예: "cube-backend PR 리뷰 3건 처리"). `commitment`보다 부담 낮은 표현.
- `evening_reflection` — 자유 텍스트. 계획 vs 실제 gap, 막힌 이유, 내일로 이월 여부

**`morning_wip_ids` 무결성 정책 (명시)**:

```
- DB 레벨 FK 없음 (JSON array 제약으로 불가)
- 애플리케이션 레벨 검증:
  get_daily_checkin(date) 반환 시 morning_wip_ids JSON을 파싱하여
  todos 테이블과 JOIN하여 존재 + 유효한 todo만 반환.
  존재하지 않거나 deleted 상태인 id는 결과의 `missing_wip_ids` 필드에 별도 수집.
- 삭제된 todo id 자체는 JSON 안에 남겨둠 (감사 가치 — "그때 이걸 WIP로 잡았었음" 기록)
```

### 기존 테이블과의 역할 구분 (혼동 방지 규칙)

| 테이블 | 축 | 주체 | 용도 |
|---|---|---|---|
| `tasks` | Retrospective | work-digest 자동 | "어제 얼마 동안 뭘 했는지" (정제 + segments) |
| `task_suggestions` | AI 제안 | life-coach 자동 추출 | "이것도 해야 함" 추천 (pending/resolved) |
| **`todos`** | **Prospective** | **사용자 수동** | **본인이 관리하는 할 일 (backlog/wip/done)** |
| `followup_chains` | Open loop | 혼합 | 막힌 채 방치 위험 있는 follow-up |

**Phase 1에서는 `todos`와 `tasks` 간 자동 연결 없음.** Loose matching (repo/category 기준)만. 자동 연결은 Phase 2+.

**Phase 1에서 `task_suggestions`는 read-only 노출** (참고용 섹션으로 아침 액션 출력에 표시). 사용자가 보고 수동으로 `todo_crud.py add` 가능. 자동 승격은 Phase 2+.

## life-coach 재구성

life-coach가 현재 **실질 dead 상태**이고 사용자가 개편을 허락했으므로, 최소 변경으로 **의미를 재정의**한다. 기존 스크립트는 건드리지 않고 SKILL.md와 문서만 재구성.

### 새 구조: 두 축으로 분리

```
life-coach 재구성 후:

축 1: 액션 (수동 invoke, 대화형, DB CRUD) — NEW
  ├─ todo-morning   — 아침 계획 (WIP 선택 + intent 기록)
  ├─ todo-evening   — 저녁 회고 (+ work-digest 실행 + reflection)
  └─ CRUD 서브커맨드 — add / list / show / move / defer / done

축 2: 분석 모드 (리포트 중심, on-demand) — 기존 유지, 의미 재정의
  ├─ daily     — 특정 날짜 HTML 리포트 (on-demand만, 자동 cron 포기)
  ├─ weekly    — 주간 리포트 (on-demand만, 자동 cron 포기)
  └─ on-demand — 특정 주제 분석
```

### 변경 요약

| 항목 | 기존 | 변경 후 |
|---|---|---|
| "모드" 용어 | daily/weekly/on-demand 3개 | "축 1 (액션)" + "축 2 (분석 모드)" 분리 |
| 자동 cron | 21시 daily, 일요일 weekly | **전부 폐기** (어차피 dead). Phase 2+ 복구 검토 |
| daily의 정체성 | 매일 자동 분석 + 리포트 | **"특정 날짜 리포트 on-demand"로 축소** |
| weekly의 정체성 | 매주 자동 주간 분석 | **"주간 리포트 on-demand"로 축소** |
| 코칭 프롬프트 | coaching-prompts.md | 그대로 유지 (분석 모드에서 계속 사용) |
| coaching_entries 테이블 | daily가 매일 저장 | on-demand 호출 시에만 저장. 빈도 감소 OK. |

### 데이터 모델 영향

- `coaching_entries` 스키마 변경 없음
- `task_suggestions` 스키마 변경 없음 (read-only 노출만 추가)
- 새 테이블 2개만 (`todos`, `daily_checkins`)

### Phase 2+ 이월

- 자동 cron 복구 (아침/저녁 액션의 텔레그램 알림)
- todo-evening이 분석 리포트 기능 흡수 (현재 daily_coach.py 로직 재사용 가능성)
- 주간 회고 자동화 (weekly + todo-weekly 통합)

## 스크립트 I/O 계약 (명시)

**원칙**:
- 스크립트 = **data I/O + business rule 검증** (순수 CLI, 대화 없음)
- Claude 세션 = **사용자 대화 + 스크립트 오케스트레이션** (LLM 담당)
- 이 경계는 기존 프로젝트 규칙(`correction-20260307-2030-no-subprocess-llm.md`)과 일치

### 신규 스크립트 3개

경로: `plugins/life-management/skills/life-coach/scripts/`

#### `todo_crud.py` — 순수 CLI

```
서브커맨드:
  add --title "..." [--done-definition "..."] [--category ...] [--priority N]
      [--project "..."] [--parent-id N] [--deadline "YYYY-MM-DD"]
      [--estimated-min N] [--quarter "2026Q2"] [--notes "..."]
      → INSERT. stdout: {"id": N}. 기본 status=backlog.

  list [--status backlog|wip|done|blocked|deferred] [--category ...]
       [--limit N] [--sort default|priority|deadline]
      → stdout: JSON array.

  show --id N
      → stdout: JSON (subtasks 포함).

  move --id N --status {wip|done|blocked|deferred} [--reason "..."] [--force]
      → UPDATE. WIP limit + Done 정의 강제 검증. 통과 시 started_at/done_at 자동.
      → 실패: stderr + exit 1. 성공: exit 0.

  defer --id N --reason "..."
      → UPDATE status=deferred, deferred_reason set.

  done --id N
      → move --status done의 별칭. Subtask 전부 done인지 확인 (부모 done 전 경고).

모든 서브커맨드 공통:
  --json 플래그 → stdout JSON
  에러 시 stderr + exit code ≠ 0
  대화 없음. 프롬프트 없음. 순수 one-shot.
```

#### `todo_morning.py` — JSON 출력 전용

```
사용법:
  python3 todo_morning.py --date YYYY-MM-DD

동작:
  1. KST 기준 해당 날짜의 데이터 수집
  2. JSON을 stdout으로 출력

출력 스키마:
  {
    "date": "2026-04-21",
    "overdue": [todo, ...],           // deadline < today AND status NOT IN (done, deferred)
    "today_due": [todo, ...],         // deadline 오늘
    "this_week_due": [todo, ...],     // deadline today+1 ~ today+7
    "current_wip": [todo, ...],       // status='wip' (어제 이월)
    "backlog_top5": [todo, ...],      // priority 규칙 기준 top 5
    "pending_suggestions": [          // task_suggestions, read-only, 최신 5건
      {"id": N, "description": "...", "suggested_date": "...", "source_type": "..."}
    ]
  }

대화 없음. 사용자 input 안 받음.
Claude 세션이 이 JSON을 읽고 대화형으로 사용자에게 보여줌 + WIP 선택/intent 받음
→ Claude가 todo_crud.py move로 WIP 전환 + db.py upsert_daily_checkin 호출로 저장.
```

#### `todo_evening.py` — work-digest 호출 + JSON 출력

```
사용법:
  python3 todo_evening.py --date YYYY-MM-DD [--skip-digest]

동작:
  1. 해당 날짜의 daily_checkins 조회 (아침 intent, morning_wip_ids)
  2. 해당 날짜의 tasks 조회. 비어있으면:
     a. work-digest Step 1-3 실행 (결정적: scanner + segments)
     b. Step 4는 LLM이 담당 (스크립트가 아님)
        → 이 스크립트는 "Step 4 필요"를 stdout.needs_llm_task_generation로 flag
        → Claude 세션이 task 생성 후 activity_writer.py update-tasks로 저장
        → 다시 이 스크립트 호출 (--skip-digest로)
     c. 실패 시: stdout.fallback=true, raw_sessions 필드 채워서 반환
  3. tasks + aim(morning_intent) loose matching:
     - repo 일치 + keyword 부분일치 → matched
     - morning_wip_ids에 포함된 todo의 title 키워드가 tasks summary에 있으면 matched
  4. 결과 JSON stdout

출력 스키마:
  {
    "date": "2026-04-21",
    "morning_intent": "...",
    "morning_wip_ids": [N, N],
    "missing_wip_ids": [],              // morning_wip_ids 중 삭제된 것
    "actual_tasks": [task, ...],        // tasks 테이블 rows
    "fallback": false,                  // true면 raw_sessions 필드 채워짐
    "raw_sessions": [...],              // fallback일 때만
    "loose_matches": [                  // wip_id별 매칭 결과
      {"wip_id": N, "matched_tasks": [...], "match_score": 0.0~1.0}
    ],
    "unmatched_actual": [task, ...],    // 계획에 없던 예정 외 작업
    "needs_llm_task_generation": false  // true면 Claude 세션이 Step 4 수행 후 재호출
  }

Claude 세션이 이 JSON 받아:
  - 사용자에게 계획 vs 실제 대조 보여줌
  - reflection 대화로 수집
  - db.py upsert_daily_checkin으로 evening_reflection 저장
  - 내일 WIP 후보 제안 (다음 날 아침 액션에 영감)
```

## 운영 규칙 (Phase 1에서 스크립트 레벨 강제)

1. **WIP limit = 2**
   - `status='wip'` todo가 이미 2개면 새 wip 전환 거부
   - 예외: `--force` 플래그로 명시적 초과 허용 (남용 방지용 stderr 로그)

2. **Done 정의 의무**
   - `done_definition IS NULL`인 todo는 `status='wip'` 진입 불가
   - 에러 메시지: "이 todo는 완료 정의가 없어. 먼저 `done_definition` 설정 필요."
   - 모호한 동사(`기획`, `초안`, `본다`, `정리`)가 title에 포함되면 stderr warning (block 아님)

3. **새 할일 → backlog 직행**
   - `add` 커맨드의 기본 `status`는 항상 `backlog`
   - WIP로 바로 들어가려면 별도 `move` 필요
   - 이유: 아침 의식 외에는 WIP 변경 금지 — 하루 중 집중 유지

4. **KST 날짜 기준**
   - 모든 `date` 필드는 KST (Asia/Seoul)
   - 날짜 경계: 00:00 KST
   - 각 스크립트 상단에서 `from zoneinfo import ZoneInfo; KST = ZoneInfo("Asia/Seoul")` 선언
   - `datetime.now(KST).strftime("%Y-%m-%d")` 패턴 사용

## Error 경로 (work-digest 실패 폴백)

`todo_evening.py`가 work-digest를 호출할 때의 실패 처리:

```
시도 순서:

1) tasks 테이블 이미 채워짐 (life-coach 기존 방식이 돌았거나 이전에 한 적 있음)
   → 그대로 사용, 정상 매칭

2) tasks 비어있음 → work-digest Step 1-3 (scanner + segments) 실행
   a) 성공 → Step 4 필요 신호 (needs_llm_task_generation=true)
      Claude 세션이 Step 4 수행 후 activity_writer.py update-tasks
      → todo_evening.py --skip-digest로 재호출
   b) Step 1-3 실패 (예: scanner 에러, 트랜스크립트 파일 없음)
      → 폴백 경로 진입

3) 폴백 경로 — tasks를 만들 수 없음
   → stdout.fallback = true
   → raw_sessions 필드 채움 (sessions 테이블에서 해당 date의 repo, start_at, end_at,
     duration_min, tag, summary 있는 것만)
   → loose_matches는 비어있음, unmatched_actual 대신 raw_sessions 기반 뷰
   → Claude 세션이 사용자에게 "work-digest 실행 실패, raw 뷰로 폴백" 보고
   → reflection만 정상 수집
   → 다음 날 아침에 재시도 권장 안내 (또는 수동 work-digest 실행)

4) sessions 테이블마저 비어있음 (매우 드문 경우)
   → stdout.fallback = true, raw_sessions = []
   → "어제 활동 기록이 없음" 안내. reflection만 수집.
```

이 정책을 `todo_evening.py`와 Claude 세션 워크플로우 둘 다에 반영.

|---|---|---|

| **daily/weekly를 on-demand로 축소** | 삭제 | 기존 스크립트 살려두면 나중에 흡수 가능. 삭제는 비가역. |
| `todos` 네이밍 | `plans` | `docs/plans/` 폴더와 용어 충돌 방지. 사용자 원래 표현 ("todo 관리") 존중. |

| **work-digest 실패 폴백 경로 명시** | 에러 시 die | MVP 신뢰성 + 사용자 경험. raw sessions로 최소 뷰 보장. |
| `parent_id` 포함 | 제외 | 지난 2주 "기획이 안 끝남"의 직접 원인 — 큰 todo를 쪼갤 수단 없음 |

이 세션에서 드러난 사용자 사고 패턴 (향후 비슷한 결정 때 참고):

- **현실 인식**: "전부 다 이번 분기에 해야 해. 이거 말고도 일이 많아" — Top 3 프레임 거부 + Backlog + WIP 2 프레임 도입의 근거가 됨.
- **도구 만들기 선호**: lean 대안(파일+대화)보다 스킬로 구현을 선택. 이유: 장기 투자 의지 + 기존 인프라(life-dashboard) 조립 가능성. 단, 이 선호 자체가 야크 쉐이빙 위험 요인이기도 해서 1주 timebox로 제약.
- **재점검 요청**: "office-hours로 구체화한 거를 brainstorming으로 재점검" — 프로세스를 이중 검증할 줄 앎. 본 Brainstorming 섹션(여러 P0/P1/P2 이슈)은 이 요청 덕분에 도출됨.
