# 산업용 대시보드 컨텐츠 설계

## 1. 운영자 시나리오 분석

대시보드를 설계하기 전에 운영자의 핵심 질문을 정의하라. 모든 컨텐츠 배치는 이 질문에 답하기 위해 존재한다.

### 모니터링 대시보드의 3대 질문 (응답 시간 순)

| 순위 | 질문 | 목표 응답 시간 | 예시 |
|------|------|---------------|------|
| Q1 | **지금 정상인가?** | < 1초 (시선만으로) | 전체 상태 표시등, 에러 카운트 |
| Q2 | **어디가 문제인가?** | < 3초 (스캔) | 에러 유닛 하이라이트, 문제 주문 표시 |
| Q3 | **왜, 어떻게 대응하나?** | < 10초 (드릴다운) | 에러 상세, 이력, 조치 가이드 |

### 시나리오 작성 규칙

- 각 화면에 대해 "운영자가 이 화면을 볼 때 먼저 알고 싶은 것" 3개를 나열
- 각 항목에 Q1/Q2/Q3 등급 부여
- Q1 항목은 항상 노출, Q2는 1클릭 이내, Q3는 드릴다운

## 2. 정보 계층 (3-Layer Rule)

모든 정보를 3개 레이어로 분류한다. **절대 4단계 이상 중첩하지 마라** — 사용자가 길을 잃는다.

### Layer 구조

```
Layer 0: 글랜스 (Glance)
  항상 보임. 스캔만으로 상태 파악.
  예: 상태 뱃지, 에러 카운트, 핵심 수치 1-3개

Layer 1: 컨텍스트 (Context)
  1클릭/호버로 노출. 판단에 필요한 맥락 제공.
  예: 유닛별 현재 상태, 주문 진행 단계, 최근 이벤트 요약

Layer 2: 디테일 (Detail)
  명시적 드릴다운으로 노출. 원인 분석/조치에 필요.
  예: 센서 이력 테이블, 캘리브레이션 로그, 상태 변경 히스토리
```

### 레이어 배치 규칙

| 규칙 | 설명 |
|------|------|
| **L0은 스크롤 없이 보여야 한다** | above-the-fold 필수. 헤더 + 메인 뷰에 배치 |
| **L1은 같은 화면에서 접근** | 패널, 탭, 인라인 확장으로. 페이지 이동 금지 |
| **L2는 접혀 있어야 한다** | `<details>`, 탭, 모달로 숨김. 기본 접힘 상태 |
| **L0에 L2 정보를 넣지 마라** | 이력 테이블, raw 로그는 L0에 두지 않음 |
| **L2가 3개 이상이면 탭으로 묶어라** | flat 나열 금지. 관련 정보끼리 탭 그룹 |

## 3. 컨텐츠 그룹핑

### 기능별 그룹 (NOT 데이터 소스별)

잘못된 그룹핑:
```
❌ "센서 데이터" / "주문 데이터" / "로그 데이터"  (데이터 소스 기준)
```

올바른 그룹핑:
```
✅ "현재 상태" / "문제 진단" / "이력 추적"  (운영자 행동 기준)
```

### 대시보드 표준 그룹

| 그룹 | 포함 항목 | Layer |
|------|----------|-------|
| **상태 요약** | 전체 가동 상태, 에러/경고 카운트, 핵심 KPI | L0 |
| **실시간 뷰** | 장비 시각화, 진행중 작업, 활성 알림 | L0-L1 |
| **최근 활동** | 최근 완료/에러 항목 요약 (5건 이내) | L1 |
| **상세 진단** | 개별 유닛/주문 상세, 센서값, 설정값 | L1-L2 |
| **이력/로그** | 시계열 데이터, 변경 이력, raw 로그 | L2 |

## 4. Progressive Disclosure 패턴

### 패턴 A: 접기 (Collapsible)
```html
<details>
  <summary>센서 이력 (12건)</summary>
  <!-- 테이블 -->
</details>
```
- **사용**: L2 테이블, 이력 목록
- **기본 상태**: 접힘
- **summary에 건수 표시**: 펼치지 않아도 데이터 존재 여부를 알 수 있게

### 패턴 B: 탭 그룹
```
[현재 상태] [이력] [설정]
```
- **사용**: L1-L2가 3개 이상인 사이드 패널
- **첫 번째 탭**: 가장 자주 보는 정보 (보통 "현재 상태")
- **탭 이름**: 2-3글자. "캘리브레이션 샘플 이력" → "캘리"

### 패턴 C: 요약→상세
```
에러 3건 [상세 보기 →]
```
- **사용**: L0에 L1 정보를 암시할 때
- **요약은 숫자 1개 + 상태**: "에러 3건", "대기 5건"
- **상세는 클릭 시 노출**

### 패턴 D: 조건부 노출
```
정상일 때: "센서 정상" (한 줄)
이상일 때: 센서값 + 임계치 + 추이 (상세 블록)
```
- **사용**: 정상 상태에서는 정보량을 최소화
- **이상 감지 시 자동 확장**

## 5. 컨텐츠 감사 체크리스트

기존 화면을 감사할 때 이 체크리스트를 사용하라.

### 정보 과부하 감지

- [ ] 한 화면에 테이블이 2개 이상 동시에 보이는가? → 탭 또는 접기
- [ ] 사이드 패널을 스크롤해야 핵심 정보가 보이는가? → L0/L1 재배치
- [ ] 같은 정보가 2곳 이상에 표시되는가? → 중복 제거, 한 곳에만
- [ ] 정상 상태에서도 경고/에러 영역이 공간을 차지하는가? → 조건부 노출
- [ ] 컬럼이 7개 이상인 테이블이 있는가? → 컬럼 축소 또는 카드 전환

### 계층 위반 감지

- [ ] L2 정보(이력, raw 로그)가 기본 펼침 상태인가? → 접기
- [ ] L0 영역에 한눈에 안 읽히는 긴 텍스트가 있는가? → 숫자/뱃지로 압축
- [ ] 드릴다운 없이 모든 정보가 flat으로 나열되어 있는가? → 그룹핑 + 접기

### 운영 효율성 감지

- [ ] "지금 정상인가?"를 1초 안에 답할 수 있는가?
- [ ] "어디가 문제인가?"를 3초 안에 답할 수 있는가?
- [ ] 에러 발생 시 조치에 필요한 정보까지 10초 안에 도달하는가?

## 6. 컨텐츠 스펙 작성

새 화면이나 리디자인 시 아래 형식으로 스펙을 작성하라.

```markdown
## [화면명] 컨텐츠 스펙

### 운영자 시나리오
- Q1: [지금 정상인가?를 어떻게 답하는가]
- Q2: [어디가 문제인가?를 어떻게 답하는가]
- Q3: [왜/어떻게 대응하는가?를 어떻게 답하는가]

### Layer 0 (항상 노출)
| 항목 | 표현 형태 | 위치 |
|------|----------|------|

### Layer 1 (1클릭 노출)
| 항목 | 트리거 | 표현 형태 |
|------|--------|----------|

### Layer 2 (드릴다운)
| 항목 | 접근 방식 | 기본 상태 |
|------|----------|----------|

### 그룹핑
| 그룹명 | 포함 항목 | disclosure 패턴 |
|--------|----------|----------------|

### 제거/통합 대상
| 현재 | 변경 | 이유 |
|------|------|------|
```

## 7. 산업용 대시보드 안티패턴

| 안티패턴 | 문제 | 대안 |
|---------|------|------|
| **정보 민주주의** | 모든 데이터를 동등하게 나열 | 우선순위 기반 계층화 |
| **테이블 만능** | 모든 데이터를 테이블로 표시 | 요약은 카드/뱃지, 상세만 테이블 |
| **패널 폭포** | 사이드 패널에 10개+ 섹션 flat 나열 | 탭 그룹 + 접기 |
| **중복 안전망** | "혹시 모르니 여기에도 표시" | 한 곳에만. 나머지는 링크 |
| **항상 상세** | 정상/이상 구분 없이 항상 상세 표시 | 정상 시 축소, 이상 시 확장 |
| **영어 혼용 제목** | "Calibration Metric", "Weight Log" | 사용자 언어로 통일 |
