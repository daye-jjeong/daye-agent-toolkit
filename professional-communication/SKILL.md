# 업무 커뮤니케이션 가이드

## 개요

소프트웨어 개발 환경에서 효과적인 업무 커뮤니케이션을 위한 프레임워크와 가이드입니다. 이해관계자에게 이메일을 쓸 때, 팀 채팅 메시지를 보낼 때, 미팅 안건을 준비할 때 — 명확하게 소통하고 신뢰를 쌓는 데 도움을 줍니다.

**핵심 원칙:** 효과적인 커뮤니케이션은 내가 얼마나 아는지 증명하는 게 아니라, 상대방이 내 메시지를 정확히 이해하게 하는 것입니다.

## 언제 쓰나

- 팀원, 매니저, 이해관계자에게 이메일 쓸 때
- 슬랙/팀즈 등 팀 채팅 메시지 작성 시
- 미팅 안건이나 회의록 준비 시
- 기술 내용을 비기술 청중에게 설명할 때
- 상태 보고서나 진행 리포트 작성 시

**키워드**: 이메일, 메일, 채팅, 슬랙, 메시지, 미팅, 회의, 안건, 상태보고, 리포트, 커뮤니케이션

## 핵심 프레임워크

### 무엇-왜-어떻게 구조

모든 업무 메시지에 적용 가능한 범용 프레임워크:

| 구성 | 목적 | 예시 |
| --- | --- | --- |
| **무엇** | 주제/요청을 명확히 | "릴리스를 1주 연기해야 합니다" |
| **왜** | 이유 설명 | "결제 처리에서 치명적 버그가 발견됐습니다" |
| **어떻게** | 다음 단계/액션 | "QA가 목요일까지 재테스트하고, 금요일에 이해관계자에게 업데이트합니다" |

### 글쓰기 3원칙

1. **명확한 제목/목적으로 시작** — 받는 사람이 즉시 무슨 내용인지 파악할 수 있어야 함
2. **글머리표와 소제목으로 스캔 가능하게** — 벽돌 같은 텍스트는 아무도 안 읽음
3. **핵심 메시지를 먼저** — 바쁜 사람은 효율을 고마워함; 결론부터

### 청중 맞춤

커뮤니케이션 전에 자문:

1. **누구에게** 쓰나? (개발 동료, 매니저, 경영진, 고객)
2. **어느 정도의 디테일**이 필요한가? (고수준 개요 vs 구현 상세)
3. **상대에게 어떤 가치**가 있나? (이것이 그들의 업무/의사결정에 어떤 영향?)

## 이메일 작성법

### 제목 공식

| 대신에 | 이렇게 |
| --- | --- |
| "프로젝트 업데이트" | "[프로젝트X] 주간 진행상황 및 다음 단계" |
| "질문" | "[질문] API 레이트 리미팅 방식 관련" |
| "참고" | "[FYI] 화요일 오후 3시 배포 예정" |

### 이메일 구조 템플릿

```
제목: [프로젝트/주제]: [구체적 목적]

OO님 안녕하세요,

[핵심 요점이나 요청을 1-2문장으로 먼저]

**배경:**
- [요점 1]
- [요점 2]

**요청 사항:**
- [구체적 액션 또는 필요한 의사결정]
- [기한이 있다면 명시]

[선택: 간략한 다음 단계나 후속 계획]

감사합니다.
[이름]
```

### 이메일 유형별 핵심

| 유형 | 핵심 요소 |
| --- | --- |
| **상태 보고** | 진행 요약, 차단 요소, 다음 단계, 일정 |
| **요청** | 명확한 요청, 맥락, 기한, 중요한 이유 |
| **에스컬레이션** | 이슈 요약, 영향, 시도한 해결책, 필요한 결정 |
| **공지/FYI** | 변경 내용, 영향 받는 사람, 필요 액션 |

**템플릿**: `references/email-templates.md` 참고

## 팀 메시징 에티켓

> **참고:** 예시는 슬랙 기준이지만 팀즈, 디스코드 등 모든 팀 메시징에 동일 적용됩니다.

### 채팅 vs 이메일 구분

| 채팅 | 이메일 |
| --- | --- |
| 짧은 답변이 오가는 빠른 질문 | 기록이 필요한 상세 문서 |
| 실시간 조율 | 이해관계자 대상 공식 커뮤니케이션 |
| 비공식 팀 논의 | 신중한 검토가 필요한 메시지 |
| 시간에 민감한 업데이트 | 여러 파트로 나뉜 복잡한 설명 |

### 팀 메시징 원칙

1. **스레드 사용** — 메인 채널을 깔끔하게; 후속 대화는 스레드에서
2. **@멘션은 신중하게** — 불필요한 알림은 금물
3. **채널 구분** — 적합한 채널에 적합한 주제
4. **직접적으로** — "PR 리뷰 부탁드립니다"가 "혹시 바쁘신가요?"보다 나음
5. **비동기 친화적** — 즉각 응답을 요구하지 않는 방식으로

### "인사만 보내지 마세요" 원칙

대신에:
```
나: 안녕하세요
나: 계세요?
나: 하나 여쭤봐도 될까요?
[대기...]
```

이렇게:
```
나: 안녕하세요 OO님 — 배포 스크립트 관련 질문입니다.
    42번째 줄에서 권한 오류가 나는데, 혹시 본 적 있으신가요?
    에러: [에러 내용 붙여넣기]
```

## 기술 vs 비기술 커뮤니케이션

### 청중별 접근법

| 청중 | 접근 방식 |
| --- | --- |
| **개발 동료** | 기술 상세, 코드 예시, 아키텍처 구체사항 |
| **기술 매니저** | 상세와 고수준 영향의 균형 |
| **비기술 이해관계자** | 비즈니스 영향, 비유, 구현보다 결과 중심 |
| **고객** | 쉬운 말, 그들에게 어떤 의미인지, 전문용어 지양 |

### 단순화 3전략

1. **큰 그림부터, 디테일은 나중에** — 사람은 "왜"를 먼저 처리하고 "어떻게"를 나중에
2. **정확성을 잃지 않으면서 단순화** — 비유 활용; 전문용어를 쉬운 말로
3. **상황 판단** — 분위기를 읽고; 질문과 참여도에 따라 조정

### 전문용어 번역 예시

| 기술 용어 | 쉬운 말 |
| --- | --- |
| "마이크로서비스 아키텍처" | "시스템을 작고 독립적인 조각으로 나눠서 각각 따로 업데이트할 수 있게 한 구조" |
| "비동기 메시지 처리" | "작업을 대기열에 넣고 백그라운드에서 순서대로 처리" |
| "CI/CD 파이프라인" | "코드를 자동으로 테스트하고 배포하는 자동화 프로세스" |
| "데이터베이스 마이그레이션" | "데이터 저장 구조를 재정리하는 작업" |

**더 많은 예시**: `references/jargon-simplification.md` 참고

## 글쓰기 명확성 원칙

### 능동태 우선

| 수동태 (지양) | 능동태 (권장) |
| --- | --- |
| "버그가 팀에 의해 발견되었습니다" | "팀이 버그를 발견했습니다" |
| "기능이 구현될 예정입니다" | "저희가 기능을 구현합니다" |
| "테스트 중 오류가 발견되었습니다" | "테스트에서 오류를 발견했습니다" |

### 불필요한 표현 제거

| 대신에 | 이렇게 |
| --- | --- |
| "현 시점에서" | "지금" |
| "~한 경우에" | "~하면" |
| "~라는 사실 때문에" | "~이므로" |
| "~하기 위해서" | "~하려고" |
| "혹시 확인 가능하실지 여쭤봅니다" | "확인 부탁드립니다" |

### "그래서 뭐?" 테스트

작성 후 자문: "그래서 뭐? 이게 읽는 사람에게 왜 중요한데?"

명확히 답할 수 없다면 메시지를 재구성해서 가치/영향을 먼저 제시하세요.

## 미팅 커뮤니케이션

### 미팅 전: 안건 작성

모든 미팅 초대에 포함해야 할 것:

1. **명확한 목표** — 이 미팅에서 무엇을 달성할 것인가?
2. **안건** — 다룰 주제와 예상 소요 시간
3. **사전 준비** — 참석자가 미리 봐야 할 것
4. **기대 결과** — 결정이 필요? 정보 공유? 브레인스토밍?

### 미팅 중: 퍼실리테이션 팁

- **시간 제한** — "이 주제는 5분만 쓰고 넘어갑시다"
- **실시간 액션 아이템 기록** — 누가 무엇을 언제까지
- **주차장(Parking lot)** — 주제 벗어난 것은 나중에

### 미팅 후: 요약 포맷

```
**미팅: [주제] - [날짜]**

**참석자:** [이름들]

**주요 결정:**
- [결정 1]
- [결정 2]

**액션 아이템:**
- [ ] [담당자]: [할 일] - 기한: [날짜]
- [ ] [담당자]: [할 일] - 기한: [날짜]

**다음 단계:**
- [후속 미팅 필요 시]
- [공유할 문서]
```

**미팅 유형별 구조**: `references/meeting-structures.md` 참고

## 빠른 참조: 커뮤니케이션 체크리스트

업무 커뮤니케이션을 보내기 전 확인:

- [ ] **목적 명확** — 받는 사람이 5초 안에 의도를 파악할 수 있는가?
- [ ] **적합한 대상** — 적절한 사람/채널에 보내는가?
- [ ] **핵심 먼저** — 주요 포인트가 앞에 있는가?
- [ ] **스캔 가능** — 글머리표, 소제목, 짧은 문단이 있는가?
- [ ] **액션 명확** — 받는 사람이 무엇을 해야 하는지 아는가?
- [ ] **전문용어 체크** — 상대방이 모든 용어를 이해할 수 있는가?
- [ ] **톤 적절** — 전문적이면서도 딱딱하지 않은가?
- [ ] **검토 완료** — 오타나 불명확한 표현은 없는가?
