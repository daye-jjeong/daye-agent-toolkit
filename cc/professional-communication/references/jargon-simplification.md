# 기술 용어 쉽게 설명하기 가이드

기술 용어를 비개발자도 이해할 수 있는 쉬운 한국어로 바꾸는 가이드입니다. 비기술 이해관계자, 고객, 타 부서 팀원과 소통할 때 활용하세요.

## 언제 쉽게 설명할까

**쉽게 설명해야 하는 상황:**

- 비기술 이해관계자 (경영진, 기획자, 영업팀)
- 고객 및 최종 사용자
- 타 부서 협업 (인사, 재무, 법무, 마케팅)
- 아직 도메인을 배우고 있는 신규 팀원
- "그게 무슨 뜻이에요?"라고 물어보는 사람

**기술 용어를 그대로 쓸 때:**

- 같은 맥락을 공유하는 개발 동료
- 기술 문서 및 설계 명세
- 코드 리뷰 및 아키텍처 논의
- 이해하기 쉬운 것보다 정확한 표현이 더 중요한 상황

## 아키텍처 & 인프라

| 기술 용어 | 쉬운 설명 |
| --- | --- |
| 마이크로서비스 아키텍처 (Microservices) | 시스템을 작고 독립적인 조각으로 나눠서, 각 부분을 따로 수정하고 배포할 수 있는 구조 |
| 모놀리식 애플리케이션 (Monolithic) | 모든 기능이 하나로 연결된 큰 덩어리 프로그램 |
| 로드 밸런서 (Load Balancer) | 여러 서버에 일을 골고루 나눠주는 교통정리 장치 |
| CDN (콘텐츠 전송 네트워크) | 전 세계 곳곳에 콘텐츠 복사본을 두어 사용자에게 더 빠르게 전달하는 시스템 |
| 컨테이너 / 도커 (Docker) | 프로그램을 실행에 필요한 모든 것과 함께 하나의 상자에 담아두는 기술 |
| 쿠버네티스 (Kubernetes / K8s) | 컨테이너화된 애플리케이션을 자동으로 관리하고 늘리고 줄이는 운영 도구 |
| 서버리스 (Serverless) | 서버를 직접 관리하지 않고 코드만 올리면 실행되는 방식. 실제 사용한 만큼만 비용 발생 |
| API (Application Programming Interface) | 서로 다른 프로그램이 정보를 주고받을 수 있게 해주는 약속된 통로 |
| 데이터베이스 마이그레이션 (DB Migration) | 데이터 저장 방식이나 구조를 새로운 형태로 재정리하는 작업 |
| 캐싱 (Caching) | 자주 사용하는 데이터를 빠르게 꺼낼 수 있는 임시 저장소에 보관하는 것 |
| 이중화 (Redundancy) | 장애가 나도 서비스가 멈추지 않도록 예비 시스템을 두는 것 |
| 페일오버 (Failover) | 주 시스템에 문제가 생기면 자동으로 예비 시스템으로 전환되는 것 |

## 개발 프로세스

| 기술 용어 | 쉬운 설명 |
| --- | --- |
| CI/CD 파이프라인 | 코드를 자동으로 검사하고 사용자에게 배포하는 자동화된 흐름 |
| 스프린트 (Sprint) | 1~2주 단위로 목표를 세우고 개발하는 반복 주기 |
| 백로그 (Backlog) | 앞으로 만들어야 할 기능과 개선 사항을 우선순위별로 정리한 목록 |
| 기술 부채 (Technical Debt) | 빠른 개발을 위해 임시로 넘긴 것들이 쌓여서 나중에 해결해야 할 숙제 |
| 리팩토링 (Refactoring) | 기능은 그대로 두고 코드 내부 구조를 더 깔끔하게 정리하는 작업 |
| 코드 리뷰 (Code Review) | 동료 개발자가 작성한 코드를 서로 검토하여 품질을 높이는 과정 |
| 풀 리퀘스트 / PR (Pull Request) | 작성한 코드 변경 사항을 본 프로젝트에 반영해달라고 요청하는 것 |
| 배포 (Deployment) | 개발이 완료된 코드를 실제 서비스에 반영하는 것 |
| 롤백 (Rollback) | 문제가 생겼을 때 이전 버전으로 되돌리는 것 |
| 피처 플래그 (Feature Flag) | 새 기능을 별도 배포 없이 켜고 끌 수 있는 스위치 |
| A/B 테스트 | 두 가지 버전을 다른 사용자에게 보여주고 어떤 것이 더 효과적인지 비교하는 실험 |

## 성능 & 안정성

| 기술 용어 | 쉬운 설명 |
| --- | --- |
| 지연시간 / 레이턴시 (Latency) | 요청을 보낸 후 응답을 받을 때까지 걸리는 시간. 체감 속도에 직접 영향 |
| 처리량 / 스루풋 (Throughput) | 시스템이 동시에 처리할 수 있는 작업의 양 |
| 확장성 / 스케일러빌리티 (Scalability) | 사용자가 늘어나도 성능을 유지하며 대응할 수 있는 능력 |
| 가동률 / 업타임 (Uptime) | 서비스가 정상적으로 작동하고 있는 시간의 비율 |
| SLA (서비스 수준 합의) | 서비스의 안정성과 품질에 대해 고객에게 약속한 기준 |
| 99.9% 가동률 ("쓰리 나인") | 1년 기준 약 8시간 정도만 서비스가 멈출 수 있다는 의미 |
| 메모리 누수 (Memory Leak) | 프로그램이 쓴 메모리를 돌려주지 않아 점점 느려지다가 결국 멈추는 현상 |
| 경합 조건 (Race Condition) | 작업 순서가 꼬여서 결과가 매번 달라지는 예측 불가능한 오류 |
| 병목 현상 (Bottleneck) | 전체 성능을 저하시키는 특정 구간. 좁은 도로에서 정체가 생기는 것과 같음 |

## 보안

| 기술 용어 | 쉬운 설명 |
| --- | --- |
| 인증 (Authentication) | "당신이 누구인가"를 확인하는 절차 (예: 아이디/비밀번호 입력) |
| 인가 (Authorization) | "당신이 무엇을 할 수 있는가"를 확인하는 절차 (예: 관리자 권한) |
| 암호화 (Encryption) | 데이터를 허가된 사람만 읽을 수 있도록 알아볼 수 없는 형태로 변환하는 것 |
| SSL/TLS/HTTPS | 인터넷으로 데이터를 주고받을 때 중간에서 누군가 훔쳐볼 수 없도록 보호하는 기술 |
| 2단계 인증 (2FA) | 비밀번호 외에 문자 인증 등 한 단계를 더 거치는 보안 확인 절차 |
| 취약점 (Vulnerability) | 공격자가 악용할 수 있는 시스템의 허점이나 약점 |
| 침투 테스트 (Penetration Test) | 전문가에게 의뢰해 실제 해킹을 시도하듯 시스템의 허점을 찾는 보안 점검 |
| 데이터 유출 (Data Breach) | 허가되지 않은 사람이 민감한 정보에 접근하는 보안 사고 |
| 방화벽 (Firewall) | 허용된 트래픽만 통과시키고 나머지는 차단하는 보안 장벽 |

## 데이터 & 데이터베이스

| 기술 용어 | 쉬운 설명 |
| --- | --- |
| SQL / NoSQL | 데이터베이스의 종류. SQL은 엑셀 표처럼 정형화된 방식, NoSQL은 더 유연한 방식 |
| 쿼리 (Query) | 데이터베이스에서 원하는 데이터를 찾아달라고 보내는 요청 |
| 스키마 (Schema) | 데이터가 어떤 구조와 형식으로 저장되는지 정의한 설계도 |
| 인덱스 (Index) | 책 뒤편의 색인처럼, 원하는 데이터를 빠르게 찾을 수 있게 해주는 목록 |
| 복제 (Replication) | 데이터를 여러 곳에 복사해두어 안전성과 속도를 높이는 것 |
| 샤딩 (Sharding) | 데이터를 여러 데이터베이스에 나눠 저장해서 대용량 처리를 가능하게 하는 기법 |
| ETL (추출-변환-적재) | 데이터를 한 시스템에서 꺼내어 정리한 뒤 다른 시스템에 넣는 과정 |
| 데이터 웨어하우스 (Data Warehouse) | 여러 곳에 흩어진 데이터를 한곳에 모아 분석할 수 있게 하는 중앙 저장소 |

## 통신 & API

| 기술 용어 | 쉬운 설명 |
| --- | --- |
| REST API | 웹에서 시스템 간 데이터를 주고받는 가장 널리 쓰이는 표준 방식 |
| GraphQL | 필요한 데이터만 골라서 요청할 수 있는 유연한 데이터 요청 방식 |
| 웹훅 (Webhook) | 특정 이벤트가 발생하면 자동으로 알림을 보내주는 기능 |
| 비동기 처리 (Async) | 작업을 백그라운드에서 실행하여 다른 작업을 멈추지 않게 하는 방식 |
| 메시지 큐 (Message Queue) | 처리할 작업을 줄 세워서 순서대로 하나씩 처리하는 대기열 |
| 실시간 (Real-time) | 새로고침 없이 즉시 반영되는 것 |
| 웹소켓 (WebSocket) | 서버와 클라이언트가 연결을 유지한 채 실시간으로 양방향 통신하는 기술 |
| 속도 제한 (Rate Limiting) | 일정 시간 안에 보낼 수 있는 요청 횟수를 제한하여 시스템을 보호하는 장치 |

## 효과적 번역 팁

### 1. 영향부터 말하기

| 이렇게 말하지 마세요 | 이렇게 말하세요 |
| --- | --- |
| "Redis 캐싱을 적용하여 DB 부하를 줄였습니다" | "자주 조회하는 데이터를 빠른 저장소에 두어 페이지 로딩 속도를 3배 개선했습니다" |
| "인프라를 쿠버네티스로 전환했습니다" | "서버 운영을 자동화하여 장애 대응 시간이 크게 단축되었습니다" |
| "모놀리스를 마이크로서비스로 분리했습니다" | "시스템을 독립적인 단위로 나누어 한 곳의 문제가 전체에 영향을 주지 않게 했습니다" |

### 2. 비유 활용하기

익숙한 일상 개념에 빗대어 설명하면 훨씬 쉽게 이해됩니다.

| 기술 개념 | 비유 |
| --- | --- |
| API | 식당에서 손님과 주방을 연결하는 종업원. 주문을 받아 주방에 전하고 음식을 가져다줌 |
| 로드 밸런서 | 식당 입구에서 빈 테이블로 손님을 안내하는 안내원 |
| 데이터베이스 인덱스 | 책 맨 뒤에 있는 색인. 내용을 처음부터 읽지 않아도 원하는 페이지를 바로 찾을 수 있음 |
| 캐싱 | 자주 쓰는 물건은 서랍 깊은 곳이 아니라 책상 위에 올려두는 것 |
| 암호화 | 편지를 잠금 상자에 넣어 보내는 것. 열쇠를 가진 수신자만 읽을 수 있음 |
| 컨테이너 (Docker) | 이삿짐을 규격화된 박스에 넣는 것. 어떤 트럭에 실어도 안전하게 운반 가능 |
| 메시지 큐 | 은행 번호표 시스템. 순서대로 번호를 뽑고 차례가 되면 업무를 처리함 |

### 4. 정확한 표현이 필요한 경우

다음 상황에서는 기술 용어를 그대로 사용하되, 간략한 설명을 덧붙이세요:

- 법적/규정 준수 관련 논의
- 보안 사고 보고
- 기술 문서 작성
- 상대방이 상세한 설명을 요청할 때

**예시:** "TLS 1.3 암호화를 적용합니다. 이것은 데이터가 전송되는 동안 보호하는 최신 보안 표준입니다."

### 5. 한국어 번역 시 주의사항

- **외래어 남용 주의:** "디플로이"보다는 "배포", "스케일 아웃"보다는 "서버 확장"처럼 한국어 표현이 있으면 먼저 사용
- **영어 병기:** 한국어 표현이 익숙하지 않을 수 있으므로, 처음 등장할 때는 영어를 괄호 안에 병기 (예: 캐싱(Caching))
- **대상에 따른 조절:** 경영진에게는 비즈니스 영향 위주로, 기획자에게는 사용자 경험 위주로 설명의 초점을 맞추기
- **정해진 번역어 사용:** 회사 내부에서 이미 합의된 번역어가 있다면 그것을 우선 사용
