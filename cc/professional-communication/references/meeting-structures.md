# 개발자를 위한 미팅 구조 가이드

소프트웨어 개발에서 자주 쓰이는 미팅의 템플릿과 구조입니다. 효과적인 미팅을 운영하고 생산적인 결과를 만들기 위해 활용하세요.

---

## 데일리 스탠드업

**소요 시간:** 최대 15분
**주기:** 매일
**형식:** 각자 3가지 질문에 답변

### 구조

팀원별 1-2분 내로 공유:

1. **어제:** 무엇을 완료했는가?
2. **오늘:** 무엇을 할 예정인가?
3. **차단 요소:** 진행을 막고 있는 것이 있는가?

### 모범 사례

- **서서 진행** (대면 시) — 짧게 유지하는 데 도움
- **활동이 아닌 성과 중심** — "회의 참석했습니다"보다 "기능 X를 완료했습니다"
- **주차장(Parking Lot) 활용** — 깊은 논의가 필요한 사안은 스탠드업 이후로 미룸
- **15분 엄수** — 시간이 되면 종료
- **차단 요소에 즉시 대응** — 누군가 막혀 있으면 도울 수 있는 사람을 바로 연결

### 피해야 할 안티패턴

- 매니저에게 보고하는 자리로 변질시키기 (팀 싱크지, 보고가 아님)
- 스탠드업 중 문제 해결 시도 (따로 시간을 잡아서 진행)
- 기술 상세에 빠지기 (페어링 세션에서 다루기)
- "보고할 게 없다"며 스킵 (짧은 업데이트라도 가치 있음)

### 한국 업무 문화 팁

- 연차가 낮은 팀원이 먼저 말하기 어려운 분위기라면, **순서를 정해놓고 돌아가며 발언**하는 방식이 효과적
- "차단 요소 없습니다"만 반복되지 않도록, **"도움이 필요한 부분"**으로 질문을 바꿔보기
- 원격 근무 시 카메라 ON/OFF 규칙을 팀 내에서 미리 합의

## 스프린트 플래닝

**소요 시간:** 1-2시간
**주기:** 스프린트 시작 시
**목적:** 스프린트 목표와 작업 범위 합의

### 구조

| 단계 | 시간 | 활동 |
| --- | --- | --- |
| 스프린트 목표 설정 | 10분 | 이번 스프린트에서 달성할 것은? |
| 백로그 리뷰 | 20분 | 우선순위별 항목 검토 |
| 추정(에스티메이션) | 30분 | 작업 규모 산정 |
| 커밋먼트 | 20분 | 팀이 스프린트 범위에 합의 |
| 가용 시간 확인 | 10분 | 연차, 회의, 기타 일정 반영 |

### 안건 템플릿

```markdown
1. **스프린트 목표** (10분)
   - PO가 스프린트 목표 제시
   - 팀원 질의응답

2. **백로그 리뷰** (20분)
   - 우선순위 순서로 항목 검토
   - 인수 조건(Acceptance Criteria) 확인
   - 의존성 식별

3. **추정** (30분)
   - 팀의 추정 방식으로 산정 (포인트, 티셔츠 사이즈 등)
   - 큰 항목은 필요 시 분해

4. **스프린트 커밋먼트** (20분)
   - 가용 시간에 맞는 항목 선택
   - 모든 팀원이 작업 내용 이해 확인

5. **마무리** (10분)
   - 스프린트 목표 및 합의된 항목 요약
   - 리스크나 의존성 메모
```

### 모범 사례

- **사전 준비 필수** — PO는 우선순위가 정해진 백로그를, 팀은 리파인먼트를 완료한 상태로 참석
- **"무엇"에 집중, "어떻게"는 나중에** — 구현 상세는 스프린트 중 별도 논의
- **집중 시간 확보** — 커밋 시 회의, 지원 업무 등 비개발 시간을 반드시 차감
- **팀이 직접 판단** — 추정과 커밋은 팀원만 결정 (매니저나 PO가 강요하지 않음)

### 한국 업무 문화 팁

- 상사가 "이것도 넣자"고 하면 거절하기 어려운 경우가 많음 — **가용 시간 데이터를 객관적으로 보여주며** 범위를 조정
- **"열심히 하겠습니다"식 커밋은 지양** — 구체적 숫자와 근거 기반으로 합의
- 플래닝 전에 슬랙/노션으로 백로그 항목을 미리 공유하면, 회의 시간을 크게 줄일 수 있음

## 스프린트 회고

**소요 시간:** 1-1.5시간
**주기:** 스프린트 종료 시
**목적:** 지속적 개선

### 구조

| 단계 | 시간 | 활동 |
| --- | --- | --- |
| 분위기 조성 | 5분 | 체크인, 톤 설정 |
| 데이터 수집 | 20분 | 피드백 모으기 |
| 인사이트 도출 | 20분 | 패턴 논의 |
| 액션 결정 | 15분 | 개선 사항 합의 |
| 마무리 | 5분 | 감사 표현, 정리 |

### 주요 포맷

#### 시작/중단/계속 (Start/Stop/Continue)

- **시작할 것:** 새로 도입하면 좋겠는 것은?
  - 예: "코드 리뷰 시 체크리스트를 도입하자"
  - 예: "배포 전 팀 전체 알림을 시작하자"
- **중단할 것:** 그만해야 할 것은?
  - 예: "금요일 오후 배포는 그만하자"
  - 예: "불필요한 회의 참석을 줄이자"
- **계속할 것:** 잘 되고 있어서 계속할 것은?
  - 예: "페어 프로그래밍을 계속 유지하자"
  - 예: "PR 머지 전 두 명 이상 리뷰 규칙을 유지하자"

#### 좋았다/배웠다/부족했다/바랐다 (4Ls)

- **좋았다 (Liked):** 무엇이 잘 됐나?
- **배웠다 (Learned):** 무엇을 새로 알게 됐나?
- **부족했다 (Lacked):** 무엇이 부족했나?
- **바랐다 (Longed for):** 무엇이 있었으면 했나?

#### 화남/아쉬움/기쁨 (Mad/Sad/Glad)

- **화남 (Mad):** 무엇이 답답했나?
- **아쉬움 (Sad):** 무엇이 아쉬웠나?
- **기쁨 (Glad):** 무엇이 좋았나?

### 안건 템플릿

```markdown
1. **분위기 조성** (5분)
   - 이번 스프린트를 한 단어로 표현한다면?
   - 지난 회고 액션 아이템 점검

2. **데이터 수집** (20분)
   - 포스트잇(또는 온라인 보드)에 각자 의견 작성
   - 시작/중단/계속 (또는 다른 포맷) 기준으로 분류

3. **인사이트 도출** (20분)
   - 공통 주제 묶기
   - 근본 원인 논의
   - 우선순위 투표 (dot voting)

4. **액션 결정** (15분)
   - 상위 1-3개 개선 사항 선택
   - 각 액션에 담당자 및 기한 배정

5. **마무리** (5분)
   - 이번 스프린트에서 감사한 점 공유
   - 다음 스프린트에서 확인할 사항 정리
```

### 모범 사례

- **안전한 공간** — 비난 없이 시스템과 프로세스에 집중
- **액션 아이템 제한** — 구체적인 개선 1-3개로 한정 (많으면 실행 안 됨)
- **후속 조치 확인** — 다음 회고 시작 시 지난 회고 액션 점검
- **포맷 변경** — 매번 같은 형식은 지루함; 다양한 포맷을 교대로 사용
- **모두 참여** — 조용한 팀원에게도 발언 기회 보장

### 한국 업무 문화 팁

- **익명 피드백 옵션 제공** — 직접 말하기 어려운 의견도 수집 가능 (Mentimeter, Google Forms 등 활용)
- **"중단할 것"에 대한 심리적 안전감 확보** — 리더가 먼저 자기 반성을 솔선수범하면 팀원이 편하게 의견을 냄
- 회고가 "감상문 발표회"가 되지 않도록, **구체적이고 실행 가능한 액션**에 집중
- 회고 결과를 노션/컨플루언스에 기록하여 팀 자산으로 축적

## 아키텍처/기술 설계 리뷰

**소요 시간:** 45-60분
**주기:** 중요한 기술적 의사결정이 필요할 때
**목적:** 구현 전에 기술적 접근 방식에 대한 피드백 확보

### 구조

```markdown
1. **맥락 및 문제 정의** (10분)
   - 어떤 문제를 해결하려는가?
   - 왜 지금인가? 무엇이 이 작업을 촉발했는가?

2. **제안 솔루션** (15분)
   - 고수준 아키텍처 다이어그램
   - 주요 컴포넌트와 각각의 역할
   - 데이터 흐름

3. **트레이드오프 및 대안** (10분)
   - 어떤 대안을 검토했는가?
   - 왜 이 접근 방식을 택했는가?
   - 무엇을 trade-off하고 있는가?

4. **질의응답 및 토론** (15분)
   - 자유 질문
   - 우려 사항 및 리스크
   - 엣지 케이스와 장애 시나리오

5. **결정 및 다음 단계** (5분)
   - 승인 / 수정 후 승인 / 재검토 필요
   - 액션 아이템 및 일정
```

### 안건 템플릿

```markdown
## [시스템/기능명] 아키텍처 리뷰

**발표자:** [이름]
**일시:** [날짜/시간]
**관련 문서:** [설계 문서 링크]

### 사전 읽기 자료
- [설계 문서]
- [관련 ADR 또는 RFC]

### 논의 사항
1. [주요 논의 포인트 1]
2. [주요 논의 포인트 2]
3. [주요 논의 포인트 3]

### 결정 필요 사항
- [결정이 필요한 항목들]
```

### 모범 사례

- **자료 사전 공유** — 설계 문서를 24-48시간 전에 배포; 미팅 시간에 읽는 것은 낭비
- **시간 제한 엄수** — 실시간 작업 세션으로 변질되지 않도록 관리
- **아키텍처에 집중, 코드는 나중에** — 구현 상세는 별도 세션에서
- **결정 기록** — 결과를 ADR(Architecture Decision Record) 또는 설계 문서에 반드시 기록

### 한국 업무 문화 팁

- 시니어 개발자의 의견에 무조건 동의하는 분위기 방지 — **"다른 의견 있으신 분?"을 명시적으로 질문**
- 설계 문서를 한국어로 작성하되, **기술 용어(API, DB, Cache 등)는 영어 그대로** 유지하는 것이 팀 내 소통에 유리
- 리뷰 결과를 **노션/컨플루언스에 ADR로 남겨** 의사결정 히스토리를 팀 전체가 공유

## 1:1 미팅

**소요 시간:** 30-60분
**주기:** 주 1회 또는 격주
**목적:** 지원, 피드백, 커리어 성장

### 구조 (유연하게 조정)

```markdown
1. **상대방의 주제 먼저** (15-20분)
   - 지금 무엇이 마음에 걸리는가?
   - 차단 요소, 고민, 성과

2. **피드백 교환** (10-15분)
   - 최근 업무에 대한 인정
   - 성장 영역 또는 코칭

3. **커리어/성장** (10-15분)
   - 개발 목표 진행 상황
   - 앞으로의 기회

4. **공유/행정 사항** (5분)
   - 조직 변동 사항 공유
   - 향후 일정 영향
```

### 좋은 질문 예시

**팀원에게 (매니저 -> 팀원):**

- "지금 가장 큰 차단 요소가 뭔가요?"
- "제가 어떻게 하면 더 잘 서포트할 수 있을까요?"
- "최근에 가장 뿌듯했던 일은 뭔가요?"
- "앞으로 더 많이 하고 싶은 업무가 있나요?"
- "팀에서 불편하거나 개선됐으면 하는 것이 있나요?"

**매니저에게 (팀원 -> 매니저):**

- "제게 필요한 것이 있으신가요?"
- "이번 스프린트에서 팀 성공을 위해 제가 어떻게 기여할 수 있을까요?"
- "제 업무에 대한 피드백이 있으신가요?"
- "팀이나 조직 차원에서 미리 알아두면 좋을 변화가 있나요?"

### 안건 템플릿

```markdown
## 1:1 미팅 - [날짜]

### [팀원 이름]의 주제
- [ ] (미팅 전 팀원이 직접 작성)

### [매니저 이름]의 주제
- [ ] (미팅 전 매니저가 직접 작성)

### 지난 액션 아이템 점검
- [ ] [지난 미팅에서 합의한 사항]

### 이번 미팅 액션 아이템
- [ ] [담당자]: [할 일] - 기한: [날짜]
```

### 모범 사례

- **상대방의 미팅, 상대방의 안건** — 팀원이 주제를 주도하게 하기
- **절대 취소하지 않기** — 꾸준함이 신뢰를 만듦
- **메모 남기기** — 논의 내용을 기록하여 다음 미팅에서 후속 조치
- **약속은 반드시 이행** — 해주겠다고 한 것은 실제로 하기

### 한국 업무 문화 팁

- 한국에서는 1:1이 "면담"이나 "평가"로 느껴질 수 있음 — **정기적이고 캐주얼한 분위기**로 진행하여 부담을 낮추기
- 처음 도입할 때는 **커피챗 형식**으로 시작하면 심리적 장벽이 낮아짐
- 팀원이 말하기 어려워하면, **"최근 업무하면서 불편한 점이 있었나요?"**처럼 구체적인 질문으로 유도
- 1:1 내용은 **비밀 유지가 원칙** — 이를 명시적으로 전달해야 신뢰 형성
- 노션 공유 문서에 안건을 미리 써놓으면, 미팅 시간을 더 효과적으로 활용 가능

## 장애 포스트모템 (비난 없는 리뷰)

**소요 시간:** 60-90분
**주기:** 중대한 장애 발생 후
**목적:** 학습하고 재발 방지

### 핵심 원칙: 비난 없는(Blameless) 접근

> 포스트모템은 범인을 찾는 자리가 아닙니다. 시스템과 프로세스의 취약점을 발견하고 개선하는 자리입니다.
> "누가 잘못했나?"가 아니라 **"왜 시스템이 이 실수를 허용했는가?"**를 질문하세요.

### 구조

```markdown
1. **타임라인 정리** (20분)
   - 언제 무엇이 발생했는가?
   - 이벤트에 대한 공통 이해 형성

2. **영향 평가** (10분)
   - 무엇이 영향을 받았는가?
   - 고객 영향, 비즈니스 영향
   - 영향 지속 시간 및 범위

3. **근본 원인 분석** (20분)
   - 5 Whys 또는 특성요인도(Fishbone) 활용
   - 트리거뿐 아니라 기여 요인까지 분석
   - "왜 이것이 감지되지 않았는가?"도 포함

4. **잘된 점** (10분)
   - 대응 과정에서 잘 작동한 것은?
   - 계속 유지해야 할 것은?

5. **액션 아이템** (15분)
   - 예방 조치 (Prevention)
   - 탐지 개선 (Detection)
   - 대응 개선 (Response)

6. **마무리** (5분)
   - 각 액션에 담당자 및 기한 배정
   - 필요 시 후속 미팅 일정
```

### 안건 템플릿

```markdown
## 장애 포스트모템 - [장애명/번호]

**일시:** [발생 일시] ~ [복구 일시]
**심각도:** [SEV1/SEV2/SEV3/SEV4]
**영향:** [영향 범위 요약]
**작성자:** [이름]

### 타임라인
| 시각 | 이벤트 |
| --- | --- |
| HH:MM | [이벤트 내용] |

### 근본 원인
- [원인 분석 내용]

- [기여 요인 2]
```

### 5 Whys 예시

```
문제: 프로덕션 서버가 30분간 다운됨

1. 왜? → 메모리 부족으로 OOM Kill 발생
2. 왜? → 메모리 누수가 있는 코드가 배포됨
3. 왜? → 해당 경로의 부하 테스트가 누락됨
4. 왜? → 부하 테스트 체크리스트에 해당 항목이 없었음
5. 왜? → 새로운 엔드포인트 추가 시 체크리스트 업데이트 프로세스가 없었음

→ 근본 원인: 새 엔드포인트에 대한 부하 테스트 프로세스 부재
→ 액션: 새 엔드포인트 추가 시 부하 테스트 항목을 자동 생성하는 CI 체크 추가
```

### 모범 사례

- **비난 금지** — 시스템과 프로세스에 집중하고, 개인을 지목하지 않음
- **선의를 전제** — 관련자는 당시 가용한 정보로 합리적 판단을 내렸다고 가정
- **액션 우선순위화** — 모든 것을 고치려 하지 말고, 가장 임팩트 있는 것에 집중
- **학습 공유** — 문서화하여 팀/조직 전체와 공유

### 한국 업무 문화 팁

- 한국 조직에서는 장애 시 **"누구 책임이냐"**로 흐르기 쉬움 — 미팅 시작 시 **비난 없는 원칙을 명시적으로 선언**
- "실수한 사람"이 아닌 **"실수를 허용한 시스템"**에 초점을 맞추는 것이 핵심
- 포스트모템 문서를 **팀 위키에 공개**하여 학습 자산으로 활용 (숨기지 않기)
- 장애 대응에 **잘한 점을 먼저 언급**하면, 이후 개선 논의가 더 건설적으로 진행됨
- 경영진 보고와 팀 포스트모템을 **분리** — 팀 내부에서는 솔직하게, 보고는 요약본으로

## 코드 리뷰 워크스루

**소요 시간:** 30-45분
**주기:** 복잡한 변경사항이 있을 때
**목적:** 중요한 코드 변경에 대한 심층 리뷰

### 구조

```markdown
1. **맥락 설정** (5분)
   - 이 변경이 어떤 문제를 해결하는가?
   - 왜 이 접근 방식을 선택했는가?

2. **고수준 설명** (10분)
   - 아키텍처/구조 개요
   - 주요 파일과 역할
   - 변경 범위 설명

3. **상세 리뷰** (20분)
   - 핵심 로직 경로 따라가기
   - 직관적이지 않은 결정에 대한 설명
   - 실시간 질의응답

4. **마무리** (5분)
   - 남은 우려 사항 메모
   - 승인 절차 합의
```

### 안건 템플릿

```markdown
## 코드 리뷰 워크스루

**PR 링크:** [PR URL]
**작성자:** [이름]
**리뷰어:** [이름들]

### 변경 요약
- [변경 사항 1줄 요약]

### 핵심 변경 파일
1. `[파일 경로]` — [이 파일의 변경 이유]
2. `[파일 경로]` — [이 파일의 변경 이유]

### 집중 리뷰 요청 사항
- [특별히 신경 써서 봐주길 바라는 부분]

### 테스트 방법
- [어떻게 테스트했는지]
```

### 모범 사례

- **PR 링크 사전 공유** — 리뷰어가 미리 훑어볼 시간 제공
- **로직에 집중, 스타일은 린터에 맡기기** — 코딩 스타일은 자동화 도구로 처리
- **작성자가 워크스루 주도** — 자신의 사고 과정을 설명
- **후속 작업 기록** — 별도 PR로 처리할 이슈는 따로 트래킹

### 한국 업무 문화 팁

- 후배 개발자가 선배의 코드를 리뷰할 때 **"이건 왜 이렇게 하셨어요?"라고 질문하기 어려움** — 워크스루에서 **작성자가 먼저 자신의 고민과 대안을 설명**하면 질문 장벽이 낮아짐
- **"이것은 틀렸다"보다 "이런 방법도 고려해볼 수 있을까요?"** 형태의 표현이 건설적인 토론을 유도
- 리뷰 코멘트에 **심각도 태그**를 붙이면 소통이 명확해짐:
  - `[필수]` — 반드시 수정 필요
  - `[제안]` — 더 나은 방법 제안, 선택은 작성자
  - `[질문]` — 이해를 위한 질문
  - `[칭찬]` — 잘한 부분 인정

## 일반 미팅 팁

### 미팅 전 체크리스트

- [ ] 미팅의 명확한 목적/목표가 정해졌는가?
- [ ] 안건을 사전에 공유했는가?
- [ ] 적합한 사람만 초대했는가? (불필요한 참석자는 없는가?)
- [ ] 사전 읽기 자료를 공유했는가?
- [ ] 소요 시간이 안건 범위에 적절한가?
- [ ] 회의실 또는 화상회의 링크가 준비됐는가?

### 미팅 중 체크리스트

- [ ] 정시에 시작
- [ ] 미팅 시작 시 목표 명시
- [ ] 각 주제별 시간 제한 적용
- [ ] 액션 아이템을 실시간으로 기록
- [ ] 주제에서 벗어나는 논의는 주차장(Parking Lot)으로 이동
- [ ] 마무리 5분 확보
- [ ] 정시에 종료 (일찍 끝나면 더 좋음!)

### 미팅 후 체크리스트

- [ ] 24시간 이내에 회의록/요약 공유
- [ ] 액션 아이템에 담당자와 기한이 지정됨
- [ ] 약속한 사항에 대한 후속 조치 진행
- [ ] 가치 없는 정기 미팅은 과감히 폐지

### 미팅 요약 템플릿

```markdown
**미팅: [주제] - [날짜]**

**참석자:** [이름들]

**주요 결정:**
- [결정 1]
- [결정 2]

**액션 아이템:**
- [ ] [담당자]: [할 일] - 기한: [날짜]
- [ ] [담당자]: [할 일] - 기한: [날짜]

**주차장 (다음에 논의):**
- [이번에 다루지 못한 주제]

**다음 단계:**
- [후속 미팅 일정]
- [공유할 문서]
```

### 효과적인 미팅을 위한 원칙

1. **미팅이 없어도 되는가?** — 슬랙 메시지, 이메일, 노션 문서로 해결 가능하면 미팅을 잡지 않음
2. **30분이면 충분한가?** — 기본 1시간이 아닌 30분으로 시작; 필요하면 연장
3. **필수 참석자와 선택 참석자를 구분** — "참고용 초대"는 회의록 공유로 대체
4. **카메라와 마이크 규칙** — 원격 미팅 시 팀 내 규칙을 미리 정해두기
5. **수요일 오후 집중 시간 확보** — 미팅 없는 시간대를 팀 차원에서 합의 (예: No-Meeting Wednesday)

### 한국 업무 문화 팁

- **"회의 위한 회의"를 피하기** — 사전에 안건이 없으면 미팅을 취소하는 문화 만들기
- 정시 퇴근을 위해 **오후 늦은 시간 미팅 자제** — 특히 금요일 오후
- 미팅 중 **발언이 특정 인원에 편중**되지 않도록, 라운드 로빈 방식이나 익명 투표 활용
- 화상 미팅 시 **채팅창 적극 활용** — 말로 끼어들기 어려운 팀원도 의견 제시 가능
- 회의록을 노션/컨플루언스에 **당일 공유**하는 습관 — "그때 뭐라고 했더라?" 방지
